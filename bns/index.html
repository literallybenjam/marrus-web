The BNS

marrus.xyz (2016) : 017.C.I.2.01

The Branching Notational System [BNS] is an organizational system under which I have organized my various projects. It is designed to facilitate a canonical and machine-readible ordering system for projects, as well as generate a series of unique identifiers with a variety of representations.

Every work in the Branching Notational System can be described by its location within a branching family tree. At the top of this tree is a single author with one or more projects, which collect works under a single unifying theme or idea. Projects are then partitioned into concepts, which are certain approaches to tackling the project. Concepts are partitioned into volumes, which are self-contained works which attempt to tackle some or all of the concept, and volumes are partitioned into versions, which are specific attempts at rendering them, with small differences in approach but the same conceptual underpinnings. Finally, versions are partitioned into drafts, which often only vary from one another in minor ways, featuring refinements and edits as a version is moved towards completion. Parts may be further described as excerpts or portions of drafts. 

Author > Project > Concept > Volume > Version > Draft > Part

The location of a work within this tree must be unique in any given implementation; that is, a work must only be describable by one combination of author, project, concept, volume, version, draft, and part.

Branch descriptions

Authors

Authors should be specified in a manner akin to that in Tag URIs; that is, with a URL or email address (henceforth referred to as a signature), and a date under which that URL or email address was definitely assigned to the author(s) in question. Although the same author could hypothetically be specified using multiple signatures and/or dates, it is preferred that a single combination be chosen, even if it is predated by some of the works described; similarly, it is preferred that such a date be as nonspecific as possible (ie., only a year). Note also that signatures should be considered case-sensitive for the purposes of determining uniqueness.

Projects

Projects are a collection of works by an author under a unifying theme or idea. Every project must have the following attributes statically defined:

  • An index, which must be a nonnegative integer.
  • A shortname, which must be a string of characters, and should only use those characters which are unreserved by the URI syntax, with the additional requirements that shortnames must not contain periods, and must contain at least one non-digit character. It is recommended that shortnames avoid using underscores and tildes, but these are allowed. Shortnames are case-sensitive.

These attributes must be unique within an author, and are immutable and not reassignable. There are no requirements for uniqueness of indices or shortnames between authors.

Concepts, volumes, versions, drafts, parts

Concepts, volumes, versions, drafts, and parts must each have a statically-defined id, which must match the form of either an index or a shortname. No two concepts, volumes, versions, drafts, or parts which share the same project, concept, volume, version, or draft, respectively, can have the same id, and ids are immutable and not reassignable. However, there are no requirements for the uniqueness of ids outside of this stipulation.

Concepts, volumes, versions, drafts, and parts may also be given labels, which help to describe them. There are no requirements on the values of labels.

Rendering

Tag URIs

Tag URIs provide a mechanism for creating a unique URI for referencing a work that is machine-readable and works with technologies such as XML. The ABNF syntax for creating a BNS Tag URI is as follows: 

tagURI = "tag:" author ":bns" [ path [ "#" fragment ] ]
author = signature "," date
signature = DNSname / emailAddress
DNSname = DNScomp *( "."  DNScomp )
DNScomp = alphaNum [ *( alphaNum / "-" ) alphaNum ]
emailAddress = 1*( alphaNum / "-" / "." / "_" ) "@" DNSname
date = year [ "-" month [ "-" day ] ]
year = 4DIGIT
month = 2DIGIT
day = 2DIGIT
path = "::" shortname *5( ":" id )
index = ( %x31-39 *DIGIT ) / "0" ; nonzero integers must not start with zero
id = index / shortname
shortname = *( noDigit / DIGIT ) noDigit *( noDigit / DIGIT )
noDigit = ALPHA / "-" / "_" / "~"

BNS Tag URIs must specify each component in order. Some examples follow:

  • tag:mail@example.com,1999:bns::p1 – Project p1.
  • tag:example.net,2006:bns – The author associated with example.net in the year 2006.
  • tag:mail@example.com,1999:bns::sam:2:1:3:28:appx – Part 'appx' of Draft 28 of Version 3 of Volume I of Concept B of Project sam.

Verbose syntax

The verbose syntax is designed to maintain the ordering of entries in a BNS implementation while providing a consistent, human-readable rendering. When applied to filesystems, the filesystem tree should reflect the tree created by the BNS, although branches which are only-children may be implied. Individual entries should follow this ABNF syntax (see the previous section for more definitions), presented with Unicode code points: 

verbose = longauthor / project / concept / volume / version
         / draft / part
longauthor = signature SP "(" date ")"
project = longindex SP shortname
concept = %x43.6F.6E.63.65.70.74 SP ( alphaID
         / quotedShortname ) [ labelSuffix ]
volume = %x56.6F.6C.75.6D.65 SP ( romanID
         / quotedShortname ) [ labelSuffix ]
version = %x56.65.72.73.69.6F.6E SP ( index
         / quotedShortname ) [ labelSuffix ]
draft = %x44.72.61.66.74 SP ( medIndex
         / quotedShortname ) [ labelSuffix ]
part = %x50.61.72.74 SP ( lowerRomanID
         / quotedShortname ) [ labelSuffix ]
longIndex = 3DIGIT / ( %x31-39 3*DIGIT )
medIndex = 2DIGIT / ( %x31-39 2*DIGIT )
alphaID = 1*%x41-5A / "0"
romanID = 1*%x58 / ( *%x58 ( %x56 / ( %x49 %x56 )
         / ( %x56 %x49 %x56 ) / 1*3%x49 ) ) / "0"
lowerRomanID = 1*%x78 / ( *%x78 ( %x76 / ( %x69 %x76 )
         / ( %x76 %x69 %x76 ) / 1*3%x69 ) ) / "0"
quotedShortname = "'" shortname "'"
labelSuffix = SP %2013 SP %x201C label %x201D
label = 1*( %xA0-D7FF / %xF900-FDCF / %xFDF0-FFEF
         / %x10000-1FFFD / %x20000-2FFFD / %x30000-3FFFD
         / %x40000-4FFFD / %x50000-5FFFD / %x60000-6FFFD
         / %x70000-7FFFD / %x80000-8FFFD / %x90000-9FFFD
         / %xA0000-AFFFD / %xB0000-BFFFD / %xC0000-CFFFD
         / %xD0000-DFFFD / %xE1000-EFFFD )

The rules for utilizing this syntax are as follows:

  • Names of authors take the form "signature (date)".
  • Names of projects take the form "NNN shortcode", where "NNN" is the project index and at least three digits.
  • Other entries begin with a string specifying the level in the tree ("Concept", "Volume", etc.), with the first letter capitalized and the remaining characters lowercase. This is followed by a space, then the id. String ids are surrounded by apostrophes, while the rending of integers varies depending on the kind of entry:
    • Concepts should be a capital letter from A to Z.
    • Volumes should be a upper-roman numeral. "IV" should be used for 4, and "VIV" should be used for 9. "L", "C", "D", and "M" should not be used. (This ensures proper ordering under alphanumeric sorting algorithms).
    • Versions should be one or more digits, with no leading zeros.
    • Drafts should be two or more digits.
    • Parts should be a lower-roman numeral, following the rules for volumes.
  • If labels have been assigned, they should be provided following the id, wrapped in left and right double-quotes, and preceded by an en-dash surrounded by spaces.

Some examples follow:

  • 092 example describes an example project with index 92.
  • example.net (2006) describes the author signified by example.net in 2006.
  • Volume XVIV – “Example volume” describes the nineteenth volume of something.

Abbreviated syntax

Abbreviated BNS syntax is as follows:

aProject = [ longauthor SP ":" SP ] longIndex
aConcept = [ aProject "." ] ( alphaID / shortname )
aVolume = [ aConcept "." ] ( romanID / shortname )
aVersion = [ aVolume "." ] ( index / shortname )
aDraft = [ aVersion "."] ( medIndex / shortname )
aPart = [ aDraft "." ] ( lowerRomanID / shortname )

This syntax follows the integer rendering rules from the verbose syntax, but cuts out the extra content. Information which can be gathered from context may be left out.

Here are some examples:

  • 015.B.I.2.03 – Draft 03 of Version 2 of Volume I of Concept B of Project 015 of some author.
  • 2.01.notes – Part 'notes' of Draft 01 of Version 2 of some work.
  • mail@example.com (1999) : 022.C.II – Volume II of Concept C of Project 022 by mail@example.com (1999)
